## The Complete Event Pipeline

### The Problem Batching Solves

Every time a user scrolls, watches a video, or skips something, that's data the algorithm needs. But if you sent one HTTP request per action, a user scrolling through 20 videos in 30 seconds = 20+ network requests simultaneously. That kills battery, wastes bandwidth, and hammers your server. **Batching collects multiple events and sends them together.**

---

## Step 1: Flutter — Events Queue Up In Memory (RAM)

```
User scrolls / likes / watches video
           ↓
EventTracker._track() is called
           ↓
Event added to _eventQueue (a List in RAM)
```

The queue lives in **the phone's RAM only** — nothing on disk, nothing on server yet. It looks like:
```json
[
  { "eventType": "view", "contentId": "42", "watchDurationMs": 8400, "completionRate": 0.87 },
  { "eventType": "skip", "contentId": "43", "watchDurationMs": 900 },
  { "eventType": "like", "contentId": "44" }
]
```

**Does it store every scroll?** No — only meaningful events:
- `view` — when user watches a video (with exact ms watched)
- `like`, `share`, `comment`, `save` — explicit actions
- `skip` — swiped past quickly
- `not_interested` — explicit rejection
- `rewatch` — looped the video
- `scroll_fast/medium/slow` — scroll velocity (new)
- `pause` — stopped scrolling for 500ms+ on content (new)
- `impression` — content appeared on screen (new)

---

## Step 2: Two Conditions Trigger a Flush (Send to Server)

**Condition A — Batch is full (10 events):**
```dart
if (_eventQueue.length >= _batchSize) {  // batchSize = 10
    _flush();  // Send immediately
}
```

**Condition B — Timer fires every 5 seconds:**
```dart
_flushTimer = Timer.periodic(Duration(seconds: 5), (_) => _flush());
```

Whichever comes first. So a fast scroller who skips 10 videos in 2 seconds sends immediately. A slow reader who watches one long video sends after 5 seconds.

**On failure:** events go back into the queue and retry next flush. Queue is capped at **200 events max** — oldest are dropped if full (prevents memory leak on bad network).

---

## Step 3: One HTTP Request — POST /api/v1/events/batch

All queued events go in a single request body:
```json
{
  "events": [
    { "userId": "5", "contentId": "42", "eventType": "view", ... },
    { "userId": "5", "contentId": "43", "eventType": "skip", ... }
  ]
}
```

The Go handler responds immediately with `{"status":"ok"}` and processes everything **in a background goroutine** so the network call returns fast.

---

## Step 4: Server — Two Storages Updated Simultaneously

For each event, the backend does **two things in parallel**:

```
Event arrives at server
         ↓
    ┌────┴────┐
    ↓         ↓
PostgreSQL   Redis
(permanent)  (temporary)
```

### PostgreSQL — Permanent Storage

```go
INSERT INTO feed_events (user_id, content_id, content_type, event_type,
    watch_duration_ms, total_duration_ms, completion_rate,
    session_id, session_position)
VALUES ($1, $2, $3, $4, $5, $6, $7, $8, $9)
```

**Every event is stored permanently.** This is the raw history — never deleted. It's used to:
- Compute UserProfile (who is this person? what do they like?)
- Build CategoryAffinity (what categories do they watch most?)
- Calculate collaborative filtering (users similar to you)
- Track win/loss momentum
- Power experiment results

### Redis — Temporary Cache (SessionState)

```go
key = "session:userId:sessionId"
// Stores: DopamineBudget, SkipCount, SkipStreak, ItemsSeen,
//         CategoriesSeen, LastEmotions, ResistanceLevel, ...
```

**This expires automatically after 30 minutes of inactivity** (the TTL). It only tracks what's happening RIGHT NOW in the current session:
- Is the user tired? (`DopamineBudget`)
- Are they skipping a lot? (`SkipStreak`)
- What categories have they seen too much of? (`CategoriesSeen`)
- What emotions have they been consuming? (`LastEmotions`)
- Are they resisting the algorithm? (`ResistanceLevel`)

Redis is used because it's in-memory and handles 50+ writes/minute per active user — PostgreSQL would choke on that volume.

## What TikTok/Instagram Actually Track

Yes — they store **every single impression**. Not just what you watch, but every piece of content that appeared on your screen, even for 0.2 seconds.

Their signal hierarchy looks like this:

```
Content appears on screen              → IMPRESSION (logged)
User scrolls past in < 500ms          → implicit rejection (weak negative)
User slows down / pauses 1-2s         → curiosity signal (weak positive)
User watches 30%+                     → interest (moderate positive)
User watches 70%+                     → strong interest
User rewatches                        → very strong interest
User likes/shares                     → explicit positive
User "not interested"                 → explicit negative
```

The most valuable data point is **"I showed it, they ignored it."** That's what builds the "don't show this" model. Without it, the algorithm only knows what you like — not what you actively don't want.

---

## What We're Missing

Currently our code does:

- `trackView` — user watched something ✅
- `trackSkip` — user swiped away ✅
- `trackScrollSpeed` — how fast they scrolled ✅ (added recently)
- `trackPause` — paused 500ms+ ✅ (added recently)
- `trackImpression` — function exists ⚠️ **but has no dwell time, and isn't wired into the feed automatically**

The gap is: **we don't automatically log every card that appeared on screen with how long it was visible.** The `trackImpression` function exists but it has to be manually called, and it doesn't record dwell time.

What we're not capturing:

```
User scrolls through 8 dance videos without stopping
→ we record: nothing
→ what we should record: 8 impressions with ~300ms dwell each
→ signal: this user doesn't want dance content
```

## Why "Shown But Ignored" is So Powerful

Right now if the algorithm shows a user 20 cooking videos and they skip all of them, we get:
- 20 `skip` events IF the video autoplayed and they swiped
- But if they scrolled the feed quickly and those videos were just thumbnails they scrolled past — **we get zero signal**

TikTok uses impression data to solve the cold-start problem for categories:
- New user, no history → show variety
- User scrolls past every sports thumbnail in < 0.5s for 3 sessions → add `sports` to `avoidedCategories`
- Without impressions, this takes weeks of explicit skips. With impressions, it takes one session.

## How to Fix It

Two things need to change:

**1. Flutter side** — the feed page needs to automatically track every card that enters and leaves the viewport with dwell time. This is a `VisibilityDetector` widget wrapping each feed item. When a card enters screen: start timer. When it leaves: log `impression` with exact ms it was visible.

**2. Backend scoring** — treat impression dwell time as a signal tier:

```
< 300ms visible   → strong implicit skip (stronger than a swipe-skip in some ways)
300ms - 1s        → weak implicit skip
1s - 3s but no watch → curiosity but didn't commit
```

And feed this into `CategoryAffinity` as a negative weight — same as we do for `not_interested` but softer.

## Storage Question

Do you store impressions permanently in PostgreSQL like other events?

**Not raw** — at scale that's billions of rows fast. TikTok's approach (and the right one) is:

- Store raw impressions temporarily (Redis, 24-48 hours)
- Aggregate them into the user profile (bump down category affinity if repeatedly ignored)
- Discard the raw impression row after aggregation

So the permanent signal isn't "user saw video #4521 for 200ms on March 5th" — it's "user's dance affinity dropped from 0.4 to 0.2 this week due to consistent ignoring." That's what matters long-term and that already lives in `user_profiles`.
