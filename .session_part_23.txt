── ASSISTANT @ 2026-04-19T17:15:56.115Z ──
**Thompson sampling** is a strategy for the classic *multi-armed bandit* problem: you have several "arms" (options) to pull, each with an unknown payoff rate, and you need to balance **exploration** (trying uncertain arms) vs **exploitation** (pulling the arm you currently think is best).

**How it works**

For each arm, keep a probability distribution over its true success rate — usually a **Beta distribution** parameterized by `(α, β)` where:
- `α` = 1 + successes
- `β` = 1 + failures

To pick an arm:
1. Sample one random number from each arm's Beta distribution.
2. Pull the arm with the **highest sampled value**.
3. Observe the reward (success/failure) and update that arm's `α`/`β`.

**Why it works**

- Arms with few observations have wide distributions → they sometimes sample high → they get explored.
- Arms with many observations have narrow distributions centered on their true rate → consistent exploitation of winners.
- Exploration naturally decays as you gather data — no epsilon to tune, no UCB constant.

**In your codebase**

`bandit.go` does exactly this for feed strategies (the `sampleBest` call in `pickAlternateStrategy`). Each strategy is an arm; each session outcome (`delta`) becomes a reward in `[0, 1]` via `(delta+1)/2`, and `updateArm` bumps that arm's posterior. The Gamma/Marsaglia-Tsang method in there is just how you draw Beta samples (Beta(α,β) = Gamma(α)/(Gamma(α)+Gamma(β))).

**Why it's better than A/B testing** for your use case: A/B splits traffic evenly even after a winner is obvious. Thompson sampling shifts traffic to winners automatically while still probing losers enough to catch regime changes.

## What I added this session & why

Think of it as **"how do you know your recommendation algorithm is actually working in production?"** That's the whole theme.

### 1. Prometheus — "a ruler"
A database that asks your backend every 15 seconds: *"How many requests did you serve? How fast? Any errors?"* It stores those numbers as timeseries.

**Why related to the algorithm:** Without this, you ship a ranker change, users get worse feeds, and you only find out when they uninstall. With Prometheus, you see **immediately** that `devf_ltr_flushes_total{outcome="error"}` is rising — your learned weights aren't saving, the algorithm is stuck with stale knowledge.

### 2. Grafana — "the display for the ruler"
Charts and dashboards. Reads from Prometheus, draws graphs. I built a dashboard with 11 panels showing: feed latency, bandit writes, LTR flushes, signal captures per minute, 5xx rate, analytics job outcomes.

**Why related to the algorithm:** When you tweak the ranker, you watch these charts to verify your change was good (p95 latency flat, engagement signals up) or bad (error rate up, revert).

### 3. Alertmanager — "the ruler screams at you"
Watches the metrics. If "5xx > 1% for 5 min" or "analytics job hasn't run in 90 min", it pages you.

**Why related:** The ranker silently failing (bandit writes all erroring, LTR never flushing) is worse than a visible crash — users get a frozen algorithm. Alerts catch this.

### 4. Docker / docker-compose — "install-button"
One command (`docker compose up`) starts Prometheus + Grafana + Alertmanager together with correct configs wired. Without Docker, you'd install each tool manually, wire ports by hand, reconfigure every restart.

**Why related:** It's just packaging. No algorithm effect, but makes the monitoring stack actually usable so the algorithm can be observed.

### 5. CI/CD (GitHub Actions) — "robot that runs your tests"
Every time you push code, GitHub spins up a Linux machine and runs `go test -race ./...`. Specifically Linux because your Windows doesn't have gcc, which `-race` needs.

**Why related:** The ranker has background goroutines (LTR flusher, analytics scheduler, bandit writer). A data race on shared state silently corrupts learned weights. `-race` catches it. The CI runs it on every change so nobody ships a race-condition regression.

## Where the algorithm can be genuinely better

Right now it's a solid single-tower linear ranker. The big gaps vs. TikTok/YouTube-grade:

| Gap | What it means | Expected impact |
|---|---|---|
| **1. No embeddings retrieval** | All candidates come from heuristic SQL (fresh / trending / following). No "semantic neighbors" of what the user actually engaged with. | Huge — this is how TikTok recommends something you've never searched for but end up loving. |
| **2. No diversity re-ranker** | After scoring, top-N is just `sort desc`. You might see 5 cooking posts in a row. | Medium — hurts session length. MMR fix is cheap. |
| **3. No position-bias correction** | LTR treats "item saw a like at position 1" and "item saw a like at position 15" as equal signals. Position 1 has 4-10× higher baseline CTR. | Medium — your LTR is slowly learning "everything is great" because top-ranked items always win. |
| **4. Single candidate source** | One pool, then score. Industry-standard is multiple retrievers (embedding-ANN, collab-filter, trending, follow-graph) merged. | Large — increases ceiling of what ranker can pick from. |
| **5. No calibration** | LTR outputs raw logits, added to other heuristic scores with arbitrary scales. Combining uncalibrated scores is noisy. | Small-medium — cleaner signal, more predictable experimentation. |
| **6. Bandit arms are hard-coded strategies** | Bandit picks between ~6 named strategies. Better: learn a soft mix weight per strategy. | Small — incremental. |
